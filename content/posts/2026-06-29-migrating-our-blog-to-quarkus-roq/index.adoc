= Migrating our blog to Quarkus Roq
wjglerum
v1.0, 2026-06-29
:title: Migrating our blog to Quarkus Roq
:lang: en
:tags: [quarkus, roq, asciidoc, static-site, migration]

This blog used to run on two repositories. One held the posts: 457 AsciiDoc
files going back to 2005. The other was a Play Framework application that, at
startup, pulled every post over the GitHub API, rendered it with AsciiDoctorJ,
and served the result dynamically.

That split was friction for everyone. You could not preview your post without
running the engine, a deploy meant restarting a JVM, and "where does this change
go" was a fair question for devs and non-devs alike. So we moved the whole thing
to https://iamroq.dev[Quarkus Roq], a static site generator with a Java soul,
and put the output on GitHub Pages. One repository, add a post, open a pull
request, get a preview, merge, done.

Here is how it went: the parts that were a pleasure, and the sharp edges that
cost us an afternoon each.

== The good parts

=== AsciiDoc stays AsciiDoc

Roq renders `.adoc` natively, so we did not convert 457 posts to Markdown. The
build never failed because of a missing AsciiDoc feature. Keeping the content
format boring was the right call.

=== One line preserved every URL

This was the most satisfying moment of the migration. By default Roq builds a
post's URL from its slugified _title_. Our old URLs were built from the
_filename_ (`/posts/2026-06-10-software-defined-vehicle-backend`), so the default
would have quietly broken every link and every search result we have.

Roq has a link-template system with placeholders. We set one template on the post
layout:

[source]
----
link: /:collection/:year-:month-:day-:name
----

`:name` is the filename with the date prefix removed, which is exactly our old
slug, and `:year`, `:month` and `:day` come from the date parsed out of the
filename. That single line reproduced all 457 existing URLs, with no per-post
edits. We confirmed it by diffing the generated paths against the live site.

=== A fast feedback loop

`quarkus dev` with hot reload, driven through the Quarkus Agent MCP, made the
find-a-problem, fix-it, re-render cycle quick. For a migration where most of the
work is discovering what breaks, that loop matters more than anything.

=== Page bundles

Each post is now a folder: `content/posts/<slug>/index.adoc` with that post's
images right next to it, referenced by bare filename. For non-developers this is
the whole mental model: one folder per post, drop your images in, done.

=== RSS and a sitemap for almost free

A two-line content file gave us back a full RSS feed at the old `/feed/` URL. The
sitemap plugin plus a one-line content file produced a 458-URL `sitemap.xml`,
something the old engine never had.

== The sharp edges

=== Build-time versus runtime configuration

The flag that triggers static generation, `quarkus.roq.generator.batch`, is a
_build-time_ property. Pass it when you run the jar and it is silently ignored
with only a warning, and the application starts as a normal server that never
generates anything. We lost time to a "hang" that was really "generation was
never going to run." Bake the flag in at package time and the build emits the
static site to `target/roq`.

=== The pure-Java AsciiDoc parser has gaps

Roq's fast, native-friendly AsciiDoc plugin uses a pure-Java parser rather than
AsciiDoctorJ. It is quick, but it is not yet a complete implementation. Rendering
all 457 posts surfaced three issues:

* *Literal blocks crash it.* Twenty-four posts threw
`NullPointerException: io.yupiik.asciidoc.model.Listing.options() is null`. The
trigger was the literal block, delimited by four dots. Converting those to listing
blocks (four dashes) renders identically and fixes it.
* *Orphaned callouts crash it too.* One post used callout markers in a source
block but explained them in prose instead of a callout list. That combination
hit the same null pointer. Adding a proper callout list after the block fixed it,
and is better AsciiDoc anyway.
* *The `video::` macro is silently dropped.* No error, the embed simply
disappears. Seven posts. We replaced each one with a passthrough HTML `iframe`.

The lesson is simple: render your entire corpus early. A spike that built all 457
posts answered the scary questions before we touched the real repository.

=== Read the source for the keys

We lost time guessing front matter keys. `permalink` and `url` are not
recognized and are silently ignored; the key is `link`. And `site.title` and
`site.description` are not configuration properties, they come from the index
page's front matter, so putting them in `application.properties` fails config
validation at build time. When the docs were thin, decompiling the extension
classes (and reading the `-sources` jars) was the fastest path to the truth.

=== `static/` is not `public/`

Files under `static/` keep a `/static/` prefix in the output. Files under
`public/` serve from the site root. We put media in `static/media` first and
every image 404'd, because the pages referenced `/media/...` while the generator
wrote `/static/media/...`. Moving media to `public/media` fixed it.

=== The generator fails fast

The generator crawls every page over HTTP and aborts the whole run on the first
non-200 response. That is excellent for catching regressions once you are clean,
but rough while you still have known-broken pages: you fix one, re-run, hit the
next. Get to zero failures and keep it there.

=== Where we drew the line: aliases

Nine of our posts have non-ASCII characters in their filenames (apostrophes,
accents, a colon). Roq slugifies those into the URL, so unlike the other ~448
posts, these nine change address. The natural fix is a redirect via the aliases
plugin, and for ASCII paths it works well, emitting static meta-refresh pages.

For non-ASCII paths it does not work in either static generation or live serving
in the version we used, and a colon in a path even breaks route registration at
startup. Rather than fight it, we accepted the URL change for those nine old
posts and documented every one of them. Pick your battles.

== Legacy content is messier than you think

Co-locating images into bundles was not a clean one-to-one move. Around 190 posts
have no images at all. Some image folders are shared between language variants of
the same post. A handful of "year in review" posts embed images from many _other_
posts. And the by-convention `background.png` is never referenced in the body. We
wrote a reference-aware tool that moved each post's own images into its bundle and
left shared assets where they were. Then there was the long tail: spaces and
percent-encoding in filenames, a Kubernetes `image:` line that looked like an
AsciiDoc macro, and a few links that were already dead. Budget for the tail.

== Would we do it again

Yes. Roq is a good fit for an AsciiDoc blog you want to make static, and its URL
template system was powerful enough to preserve our scheme exactly. The headline
advice: render your whole corpus in a throwaway spike on day one, learn which
properties are build-time before you write your CI, and do not lean on aliases for
non-ASCII URLs yet.

All 457 posts now generate with zero failures, every modern URL is unchanged, and
adding a post is a folder and a pull request.
